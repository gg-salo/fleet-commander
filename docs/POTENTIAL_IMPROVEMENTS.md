Cross-session awareness (highest impact)
Right now your sessions are fully isolated — that's a feature for simplicity but a liability at scale. When you have 10+ agents working on the same codebase simultaneously, the interactions between them matter more than any individual session.
The most impactful addition would be a conflict prediction layer sitting inside the lifecycle manager. Before spawning a session, it would scan the issue description and (optionally) run a lightweight analysis of which files/modules are likely touched, then cross-reference against active sessions. This doesn't need to be perfect — even a heuristic based on directory overlap between worktrees' changed files (just git diff --name-only on active branches) would catch the obvious collisions. The orchestrator could then either sequence conflicting work, or at minimum warn the second agent that another session is modifying overlapping files and to pull before pushing.
A step further: session grouping. Your planning workflow already creates multiple tasks from a single feature. Those tasks often have implicit ordering — you can't build the API endpoint before the database migration. Right now dependency tracking exists in the plan but doesn't flow into spawn ordering or lifecycle management. If the lifecycle manager understood "session app-3 depends on app-2 being merged," it could hold app-3 in a waiting_dependency state and auto-spawn when app-2 merges. That turns Fleet Commander from parallel execution into intelligent pipeline orchestration.
Reaction intelligence
Your reaction system is rule-based with fixed escalation thresholds. That works, but it leaves a lot on the table. Consider three additions:
First, error classification. When CI fails, the raw error output gets forwarded to the agent. But not all CI failures are equal. A lint error is almost always fixable by the agent. A flaky integration test might not be the agent's fault at all. A type error in a file the agent didn't touch suggests a merge/rebase issue. If you classify the CI failure before reacting, you can route differently — auto-retry for lint, escalate immediately for flaky tests, trigger a rebase for upstream conflicts. The classification could be as simple as regex pattern matching on the error output, or you could use a lightweight LLM call to categorize.
Second, reaction success tracking. You already track attempt counts per (sessionId, reactionKey). Extend that to track outcomes — did the agent's CI fix attempt actually result in passing CI on the next poll? Store that as a simple success/failure per reaction execution. Over time, you build a dataset: "for this project, send-to-agent on CI failures succeeds 70% of the time on attempt 1, 30% on attempt 2, and almost never on attempt 3." That data lets you tune escalation thresholds dynamically rather than using static defaults. Even exposing it in the dashboard as "reaction effectiveness" metrics would be valuable for teams.
Third, context-enriched reactions. When you send a CI failure to the agent, you're sending the raw error. But you could enrich it significantly. Which files did the agent change? What was the last commit message? Is this the same error as last time or a new one? Has another session recently merged something that could cause this? Richer context means the agent fixes the problem faster, which directly reduces cost and cycle time.
Orchestrator evolution
The orchestrator agent is currently a coordinator — it knows the CLI commands and manages workers. But it could become much more strategic:

Workload balancing. If you have 15 issues to process and you know from experience that your CI pipeline gets congested above 8 concurrent PRs, the orchestrator should batch intelligently rather than spawning everything at once.
Priority inference. Not all issues are equal. If the orchestrator can read issue labels, milestone assignments, or even just the conversation in the issue, it can prioritize higher-impact work. Combine this with cost tracking and you get "this agent has already spent $12 on a low-priority issue with no PR yet — should we kill it?"
Post-merge sequencing. After a PR merges, other active sessions may need to rebase. The orchestrator should proactively trigger rebases rather than waiting for conflicts to surface during CI.

Resilience and recovery
You have session restore but the recovery story could be stronger:

Checkpoint-based recovery. Before each major lifecycle transition (working → pr_open, ci_failed → reaction sent), snapshot the session state. If the orchestrator crashes mid-reaction, it can resume from the last checkpoint rather than replaying from scratch.
Orphan detection. If the orchestrator process dies, tmux sessions keep running. On restart, the system should reconcile what's actually alive in tmux with what metadata says should exist. You partially have this, but making it bulletproof matters when people are running this overnight.
Stale worktree cleanup. Worktrees from killed sessions can accumulate. A periodic garbage collection pass (already partially in session cleanup) that also prunes orphaned git worktrees would prevent disk bloat.

Prompt optimization loop
This one's subtle but potentially high-leverage. Your 3-layer prompt system is static — base + config + user rules. But you're generating data about what works. If agents consistently fail on certain types of tasks with certain prompts, that's signal. A lightweight feedback mechanism where the orchestrator agent (or a dedicated analysis pass) reviews failed sessions and suggests prompt adjustments would create a self-improving system. Even just logging "session failed after 3 CI retries, agent was claude-code, issue type was refactoring, prompt included X rules" gives you the data to iterate manually.
